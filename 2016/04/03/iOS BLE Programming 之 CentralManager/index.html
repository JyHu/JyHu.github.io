iOS BLE Programming 之 CentralManager

  • BLE4.0
  • CoreBluetooth
  • iOS

我的上一份工作的公司是做智能体温计的，硬件设备通过BLE4.0来进行体温数据的传输，在此过程中就需要接触BLE相关的东西，于是我就开始了一段摸索与踩坑的过程，不过，通过apple的官方文档还有Google，我还算是小有收获吧。
现在离开上一家公司有将近大半年的时间了吧，还是决定记录一下，毕竟好记性不如烂笔头嘛，虽然现在不做BLE了，万一以后啥时候需要呢？
BLE4.0(一种近距离低功耗无线数据传输协议)，由蓝牙联盟制定(Bluetooth.com)。
在苹果iOS设备中，从iPhone 4s、iOS5.0开始支持。

CentralManager ：蓝牙数据通信的中心设备，也可以成为客户端，响应的服务端-CBPeripheralManager在这里(传送门)。

一、BLE中的几个概念的解释

  • CBCentralManager
    蓝牙设备的管理中心，用来管理所有的蓝牙设备、连接请求、数据包管理等，每一次的连接都必须有一个central，对所有设备的发现、连接等等操作都需要它。
    它本身并没有几个属性，最重要的只是一个用做状态判断的state只读属性。但是它有一些协议方法，是必须实现的，而且BLE的操作大多是通过delegate来进行交互。

  • CBPeripheral
    扮演着一个远程端的蓝牙设备的角色，central所管理的就是它。
    它有一些只读属性，用做判断，其中重要的一个是UUID，它唯一的标识着一个peripheral（远程端的BLE智能硬件设备），由开发者写在硬件的固件（firmware）中。
    还一个重要并且必须的是services（NSArray类型，服务，下面介绍），里面包含了一个peripheral所包含的所有服务。
    当然必须的是，它还有一些必须实现的协议方法需要实现。

  • CBService
    服务，是每个peripheral必备的。每个service都用来表明peripheral所能提供的服务，比如温度计服务、固件信息服务等等。
    而且每个service都有一个UUID来标识着它的唯一性，其中16位的UUID只有蓝牙联盟指定规定的一些service时可以使用，比如0x180A设备信息服务、0x1809体温计服务等等，开发者自己只能使用128位的，比如0xF000FFC1-0451-4000-B000-000000000000。当然了，你也可以使用16位的。但是，如果你们的硬件设备想得到蓝牙联盟的认证，那就不可能了。
    而每个service又包含有多个characteristic（下面介绍）。

  • CBCharacteristic
    特征，是每个service必备的。每个characteristic都标识着service的一个特征值，比如180A硬件服务的2A26固件版本号、2A25串号等等。所以，我们读取peripheral的数据或者操纵peripheral的数据发送等等最后都是根据characteristic来具体执行与表示的

  • RSSI
    BLE信号强度，用负数表示，数值越小，表示信号衰减越厉害，工厂在生产蓝牙相关硬件设备的时候就可以以这个为标准来判断设备是良品还是次品。
    在iBeacon中，这个可以用来判断距离的远近，这个以后再说。

2、初始化central

有两个初始化方法，根据需要选择就行。

  • 方法一
    self.centralManager = [[CBCentralManager alloc] initWithDelegate:self queue:nil options:nil];

    第一个参数delegate，是必须设置的
    第二个参数queue，有事件发生的时候将要进行处理的线程，如果没有设置，将推送到主线程中
    第三个参数option，dictionary类型，其中的key例如：

    CBCentralManagerOptionShowPowerAlertKey
    CBCentralManagerOptionRestoreIdentifierKey

  • 方法二
    self.centralManager = [[CBCentralManager alloc] initWithDelegate:self queue:nil];

    参数说明同上

3、唯一的一个必选(require)的协议方法

  • central.state是一个用来判断蓝牙状态的枚举类型(CBCentralManagerState)：

    CBCentralManagerStateUnknown : 未知的状态
    CBCentralManagerStateResetting : 系统连接刚刚断开，正在重新连接中
    CBCentralManagerStateUnsupported : 当前设备不支持BLE4.0
    CBCentralManagerStateUnauthorized : 用户没有允许当前设备的蓝牙授权
    CBCentralManagerStatePoweredOff : 设备的蓝牙关了，需要提示用户打开蓝牙
    CBCentralManagerStatePoweredOn : 蓝牙状态正常，可以扫描蓝牙外设了

  • scanForPeripheralsWithServices:options: 开始扫描蓝牙外设

    第一个参数为services UUIDs，如果设置了， central就只会接收发送的数据包中包含有设置的这些服务的peripheral，如果没有设置，则会接收所有的设备发送的数据包并尝试进行连接
    第二个参数为一些负数连接操作的设置，其中所有的可以用参数都包含在 <CoreBluetooth/CBCentralManagerConstants.h>中，例如：
    CBCentralManagerScanOptionAllowDuplicatesKey
    CBCentralManagerScanOptionSolicitedServiceUUIDsKey
    注意：如果设置了后台扫描属性的话，第一个service UUID必须设置，而且第二个参数中设置的内容将会无效

4、当扫描到设备以后，可以连接了

5、central成功连接外设peripheral

此时可以保持连接等待后续操作，当然也可以直接继续下一步操作，比如读取peripheral的RSSI值：[self.peripheral readRSSI];。
或者搜索当前连接外设的服务：[self.peripheral discoverServices:nil];，其参数为一个CBUUID数组，如果设置了，只会搜索设置好的这些service，否则会搜索所有的service。
注意 ： 此处连接上的peripheral必须保持强引用，而且每个central都可以连接多个外设，但是没有查到具体可以连接多少，不过，之前测试的时候，同时连接十个也都成功了。

8、当搜索到peripheral的service后

这时候可以做两件事：

  • 继续下一步操作，读取service的characteristic

    [peripheral discoverCharacteristics:nil forService:service];
    其中第一个参数为CBUUID数组，如果设置了就只会搜索这些设置了得characteristic，否则会搜索所有的characteristic。

  • 读取包含的service

    [peripheral discoverIncludedServices:nil forService:service];
    如果当前service还包含有子service的话，可以通过这个方法来继续读取。
    当然，有没有子service这个是硬件方在开发的时候设置的，在iOS端可以通过service的UUID来判断，so，这需要iOS开发者与硬件开发者沟通好

9、当连接的peripheral的CBService改变的时候，这时候需要重新搜索一下service

10、当搜索到服务中包含的服务的时候，当然是需要继续去搜索其中的characteristic

11、当搜索到characteristic的时候

这时候可以做以下几件事

  • 直接读取characteristic中所包含的数据:

    [self.peripheral readValueForCharacteristic:characteristic];
    当然这时候的读取只适合在发现characteristic的时候就已经包含了需要得到的数据的时候，比如获取硬件的固件版本号、固件的软件版本等等

  • 通过设置监听的方法来订阅数据的变动：

    [self.peripheral setNotifyValue:YES forCharacteristic:characteristic];
    这种方法适合数据经常变动或需要对数据进行持续观察的characteristic，比如每隔一秒硬件体温计设备都会发送一个体温数据，这时候iOS设备端是没法知道体温计什么时候发送这个数据广播包，所以需要通过订阅notify的方式来实现。

  • 也可以直接读取characteristic的一些描述属性：

    [self.peripheral discoverDescriptorsForCharacteristic:characteristic];

12、当设置notify的characteristic的远程端（peripheral）有数据变化的时候，会触发这个协议方法，然后就可以像上一步一样读取数据了

注意 : characteristic.properties是CBCharacteristicProperties的枚举类型，其中包含着characteristic的可执行或者可进行操作的属性，例如可读、可写等等。

17、数据的写入

  • characteristic数据的写入
    比较常用的方法，用来对外设进行一些指令操作、数据写入等。

写入的value为NSData类型，数据组织方式需要跟硬件端进行约定。
第三个参数，是为了设置是否需要写入后的回复，如果需要的话，在写入数据成功以后就会通过以下协议方法通知你：

peripheral:didWriteValueForCharacteristic:error:

  • descriptor数据的写入
    不算常用的方法，写入一些描述性的数据。

    [peripheral writeValue:[NSData data] forDescriptor:descriptor];
    写入的数据Data这个是跟硬件方商量好的，按照规定好的方式来组织数据，当写入成功以后会通过以下的协议方法来通知你

    peripheral:didWriteValueForDescriptor:error:

三、剩下的一些centralManager相关的方法

  • retrieveConnectedPeripheralsWithServices:
    检索已经连接到系统的蓝牙设备，其参数为CBUUID数组，从peripheral中取得。
  • cancelPeripheralConnection:
    取消一个连接着的peripheral，这时候就会走centralManager:didDisconnectPeripheral:error:这个协议方法。
  • stopScan
    停止扫描外设。

四、后记

啰啰嗦嗦的写了这么一大长篇，如果没写错的话，写的应该都是代理方法，而且所涉及到的两个类的代理方法应该都写全了。
因为CoreBluetooth是苹果高度封装的一个BLE操作的类库，提供的操作和读取的方法大多是通过协议方法来完成和开发者的交互的，而一些trigger的方法则相对来说就比较少了。
