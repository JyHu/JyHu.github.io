Multipeer Connectivity 近场数据传输的实现

发表于 2016-05-06   |  

引用一下苹果官方的文字(About Multipeer Connectivity)：

The Multipeer Connectivity framework provides support for discovering services provided by nearby iOS devices using infrastructure Wi-Fi networks, peer-to-peer Wi-Fi, and Bluetooth personal area networks and subsequently communicating with those services by sending message-based data, streaming data, and resources (such as files).

总结一句话就是：Multipeer Connectivity就是通过Wi-Fi、P2P、BLE等近场局域网络实现数据传输的一种技术。

一、MultipeerConnectivityHeaders

下面是MultipeerConnectivity.framework框架中所有的头文件：

  • MCError:里面是错误的状态码。
  • MCPeerID:表明一个用户，不管是广播者还是接受者都必须有的。他只有一个只读属性，用于分辨不同的用户。
  • MCSession:一个会话通道，提供与周边设备进行数据通信的支持。如果创建了session，可以邀请别的用户加入会话，同时，也能通过搜索加入别的会话-session。
  • MCNearbyServiceAdvertiser:通过一个特定的服务名称向附近的用户广播，告知想要加入他们的会话中。
  • MCAdvertiserAssistant:提供MCNearbyServiceAdvertiser一样的功能，不过它提供了一个请求用户确认是否加入会话的接口，如果不想用系统的提示的话，可以通过这个接口来封装自定义的提示界面。
  • MCNearbyServiceBrowser:用于通过指定的服务名称搜索可以加入的会话，需要自己定义搜索到的设备列表界面，以供用户进行选择加入。不过也能通过下面的这个类来提供系统封装的可视界面。
  • MCBrowserViewController:封装了MCNearbyServiceBrowser的功能，并提供了一个可视界面供用户操作。

通过上面对于类的说明可以知道，在MultipeerConnectivity近场通信中有两种身份，一个是向外广播需求想要加入会话的身份，一种是扫描外面广播的需求邀请加入会话的身份，当然，在写程序的时候也能同时拥有两种身份，我们就先不提，先拆分两种身份进行说明。

两者都需要使用到的类 搜索需要的类 广播需求的类
MCPeerID MCNearbyServiceBrowser MCNearbyServiceAdvertiser

2、初始化会话

有了用户身份，在通讯的过程中还需要有“有人”来提供会话的通道，用来进行数据的交互，于是就需要session:

这个类有两种初始化方式，其实也算是一种：

  • myPeerID:本身的身份，每个session包含的peerID算上连接上的和自身的，总共只能有8个。
  • identity:包含着一些列的验证信息，用于让其他想要连接的用户进行安全鉴定。数组中的元素是SecIdentityRef结构体，详情请见<Security/SecIdentity.h>
  • encryptionPreference:数据加密的类型，默认的就是required。

session有一些只读的属性，看看就能明白什么意思，还有三个重要的写数据的方法六个重要的协议方法，下面拆开分不同的情景来描述。

三、搜索广播和发送广播

搜索和发送广播各有两种方式，可以实现，使用其中任意的一种即可。

1、这里先说第一个协议方法

暂且将它取名为SessionP1(session的第一个protocol方法):

它的作用是通知与用户(peerID)的连接状态(state)改变，主要是刷新相关页面的可操作状态(比如页面上只有在成功连接到指定用户设备后才可以点击，当然了，如果无需页面的话那就无所谓了)、通知用户状态改变、数据流的清理等操作。

2、一个重要的参数-serviceType

用于向外广播的服务的名称，因为同时可能有多款APP的很多用户在同时进行广播和搜索，如果没有一个条件来限制他们搜索的范围和用户类型，这样会很乱。
serviceType就像一个过滤器，在搜索和广播的过程中，只能发现相同服务的用户设备，而且这个参数是必须设置的。

需要注意的是，它不能够随意的去写，它有有以下两个要求：

  1. 必须为1~15个有效长度
  2. 只能使用ASCII小写字母和连字符

比如：auu-service、alipay-service、mxyc-service等等

当发现了相同服务下的用户设备的时候，就会走这个协议方法，此时就能使用invitePeer:toSession:withContext:timeout:方法来邀请发现的这台设备加入当前的会话，如果用户同意了，这时候SessionP1方法就会执行了，更新连接状态，然后就能发送数据了。
